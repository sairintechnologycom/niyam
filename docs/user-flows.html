AGENTOPS · USER FLOWS

How people use Niyam

Niyam is not another coding agent. It is the control plane around Claude, Codex, Gemini, and friends — rules in .niyam/, agents do the coding, evidence goes out the door.

1. Big picture — who does what

Architecture

You configure one source of truth. Niyam projects it into each AI runtime, enforces policy while agents work, and packages audit evidence when you ship.

4. Day-to-day interactive development

Most common

Human + one agent. Niyam provides methodology (TDD skill), path freezes, and command deny lists.

sequenceDiagram
  autonumber
  actor Dev as Developer
  participant Agent as Claude / Codex
  participant Niyam as Niyam policies
  participant Repo as Git repo

  Dev->>Niyam: niyam sync (when rules change)
  Dev->>Agent: /implement "add rate limiting"
  Agent->>Niyam: Load agents, skills, policies
  Agent->>Repo: Write tests first (TDD)
  Agent->>Repo: Implement feature
  Niyam-->>Agent: Block denied cmds / frozen paths
  Dev->>Agent: /review
  Agent->>Repo: Security + quality pass
  Dev->>Agent: /ship
  Agent->>Niyam: Validate + evidence readiness
  Dev->>Repo: Open PR
Prefer slash commands over free-form “just fix it”: /implement → /review → /ship. High-risk areas (auth, payments, infra, secrets, migrations, CI) still need a human.

5. Autonomous mission flow

Batch / multi-step

For migrations and multi-task goals: plan a DAG of task contracts, approve, execute in isolation, validate, review, report.

flowchart TB
  G["Goal / requirements file"] --> PL["niyam mission plan"]
  PL --> VP["validate-plan"]
  VP --> AP{"Human approval"}
  AP -->|approve| ST["mission start"]
  AP -->|reject / replan| PL
  ST --> EX["Execute tasks\noptional worktrees · parallel"]
  EX --> VA["Validate\ntests · lint · security"]
  VA -->|fail| RT["Retry / replan"]
  RT --> EX
  VA -->|pass| RV["Reviewer agent\ndiff + evidence"]
  RV --> RP["mission report\nreadiness score"]
  RP --> PR["PR / merge decision"]

Task contracts make autonomy deterministic: allowed/blocked files, acceptance criteria, validation commands, risk level, preferred executor.

6. Governance & readiness

Before merge

Five pillars feed one evidence pack and a numeric GO / NO-GO decision.

8. Choose your interaction mode

Pick the right loop

Same control plane — different intensity of autonomy and human involvement.

Interactive

Daily feature work with slash commands.

/implement · /review · /ship

Mission

Multi-step goals with plan approval.

niyam mission · niyam run

LoopOps

Budgeted plan → implement → evaluate cycles.

niyam loop run …

Observe

Learn risk without hard blocks first.

niyam guard run

Hard governance

Team/enterprise enforcement + CI gates.

guard enable · ci verify

Control Room

Supervised browser/tasks + portal UI.

workspace · portal · dashboard
